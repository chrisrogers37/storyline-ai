## Context

**Finding #4 (HIGH):** drizzle-orm versions prior to 0.45.2 contain a SQL injection vulnerability tracked as [GHSA-gpj5-g38j-94v9](https://github.com/advisories/GHSA-gpj5-g38j-94v9). The vulnerability exists because SQL identifiers (table names, column names) are not properly escaped, allowing an attacker to inject arbitrary SQL through identifier positions.

The landing site (`landing/`) depends on `drizzle-orm` at version `^0.45.1`, and the lock file pins exactly `0.45.1` -- a vulnerable version.

### How drizzle-orm is used in the landing site

The landing site has a minimal database layer with only 3 files that interact with drizzle-orm:

1. **`landing/src/lib/schema.ts`** -- Defines a single table `waitlist_signups` with static column names (`id`, `email`, `name`, `instagram_handle`, `content_count`, `created_at`, `notified_admin`, `invited_at`, `notes`). All identifiers are hardcoded string literals; none come from user input.

2. **`landing/src/lib/db.ts`** -- Creates the Neon HTTP database connection via `drizzle(sql, { schema })`. No dynamic identifiers.

3. **`landing/src/app/api/waitlist/route.ts`** -- The only endpoint that writes to the database. It calls `getDb().insert(waitlistSignups).values({ email })` where `email` is a user-provided string. However, `email` flows into a **value** position (parameterized), not an **identifier** position. The table name (`waitlistSignups`) and column name (`email`) are both static references from the schema definition.

**Exploitability assessment:** The current codebase does not pass user input into identifier positions (table names, column names, schema names). The SQL injection risk is therefore **theoretical rather than actively exploitable** in this specific codebase. However, the vulnerable version must still be updated because:
- Future code changes could introduce dynamic identifiers
- Defense-in-depth requires patching known vulnerabilities
- `npm audit` will continue to flag the project as insecure
- The fix is trivial with no breaking changes

## Detailed Implementation Plan

### Step 1: Update the version specifier in package.json

**File:** `landing/package.json` (line 18)

**Before:**
```json
    "drizzle-orm": "^0.45.1",
```

**After:**
```json
    "drizzle-orm": "^0.45.2",
```

**Why `^0.45.2` and not an exact pin:** The existing project convention uses caret ranges for most dependencies (e.g., `"clsx": "^2.1.1"`, `"lucide-react": "^0.575.0"`). Changing to `^0.45.2` ensures the minimum installed version is the patched release while following the existing convention. The caret allows patch updates within the 0.45.x range.

**Note on drizzle-orm versioning:** drizzle-orm follows semver. The 0.45.1 -> 0.45.2 bump is a patch release containing only the security fix for identifier escaping. There are no breaking API changes, no new required configuration, and no schema migration changes between these versions.

## What NOT To Do

1. **Do NOT run `npm audit fix --force`.** This can upgrade packages across major versions (e.g., Next.js 16.1.6 -> 16.2.3) which is out of scope for this phase and may introduce breaking changes. Only update drizzle-orm.

2. **Do NOT update drizzle-kit in this phase.** The current `drizzle-kit@^0.31.9` is compatible with `drizzle-orm@0.45.x`. Updating drizzle-kit is unnecessary and adds risk.

3. **Do NOT modify any `.ts` or `.tsx` source files.** This is a dependency-only update. The API surface of drizzle-orm is unchanged between 0.45.1 and 0.45.2.

4. **Do NOT add raw SQL queries or dynamic identifiers.** The current codebase correctly uses static schema definitions. If future features need dynamic column selection, always use drizzle-orm's type-safe query builder rather than string interpolation.

5. **Do NOT delete `package-lock.json` and regenerate from scratch.** This would change versions of all dependencies, not just drizzle-orm, making the PR noisy and risky. Use `npm install` which performs a targeted update.

6. **Do NOT run `npm update` (without arguments).** This updates ALL packages matching their semver ranges, which is broader than intended. Edit `package.json` first, then run `npm install` to update only drizzle-orm.

7. **Do NOT confuse this with the Python backend.** The Python backend (`src/`) uses SQLAlchemy, not drizzle-orm. This fix is exclusively for the Next.js landing site in `landing/`.
