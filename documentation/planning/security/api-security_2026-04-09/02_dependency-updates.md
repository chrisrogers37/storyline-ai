## Context

The security audit identified 5 findings related to outdated dependencies with known CVEs. Two HIGH-severity CVEs in the `cryptography` package affect token encryption and OAuth flows used in production. A MEDIUM-severity CORS bypass in `starlette` could expose authenticated API endpoints. A MEDIUM-severity file replacement vulnerability exists in `requests`. The landing site has 5 moderate Next.js vulnerabilities and multiple transitive dependency issues.

These are version bump changes with no business logic modifications. The risk comes from potential breaking changes in upstream libraries, not from our code changes.

## Detailed Implementation Plan

### Step 1: Update Python Dependencies in `requirements.txt`

All changes are in `/Users/chris/Projects/storyline-ai/requirements.txt`.

#### 1a. Update `cryptography` (Finding #3 -- HIGH)

**Line 38 -- current:**
```
cryptography>=41.0.0
```

**Change to:**
```
cryptography==46.0.7
```

**Why `46.0.7` and not `46.0.6`:** The research shows 46.0.6 as the minimum fix for CVE-2026-34073, but 46.0.7 is the latest available patch release. Pinning to exact version (`==`) instead of floor (`>=`) prevents future drift and aligns with the project's goal of deterministic builds. This also addresses the dependency hygiene concern about inconsistent pinning.

**Breaking change risk:** LOW. This is a patch version bump within the same minor (46.0.x). The `cryptography` library maintains backward compatibility within patch releases. The project uses `cryptography` for token encryption (`src/services/token_encryption.py`) -- no API changes occurred between 46.0.3 and 46.0.7.

**CVE verification after update:**
- CVE-2026-26007 (private key leakage): Fixed in 46.0.5. Version 46.0.7 >= 46.0.5. Resolved.
- CVE-2026-34073 (DNS name constraint bypass): Fixed in 46.0.6. Version 46.0.7 >= 46.0.6. Resolved.

#### 1b. Update `fastapi` (Finding #6 -- MEDIUM, fixes starlette transitively)

**Line 34 -- current:**
```
fastapi>=0.109.0
```

**Change to:**
```
fastapi==0.115.12
```

**Why `0.115.12`:** FastAPI 0.115.12 is the latest release in the 0.115.x line and depends on `starlette>=0.40.0,<0.46.0` which pulls in starlette 0.45.3+ (the fix for CVE-2026-33010 was backported to the 0.x line). We do NOT jump to FastAPI 1.0+ because FastAPI 1.x depends on starlette 1.x which is a major version change with potential breaking changes in ASGI middleware, exception handlers, and routing internals.

**IMPORTANT:** Verify the exact starlette version pulled in after install:
```bash
./venv/bin/pip show starlette
```
Confirm the installed version is >= 0.45.3 (the version that fixes CVE-2026-33010).

**If starlette is still < 0.45.3 after installing fastapi 0.115.12**, explicitly add starlette to `requirements.txt`:
```
starlette==0.45.3
```

**Breaking change risk:** MEDIUM. The project currently has fastapi 0.129.0 installed (according to the research), which is *higher* than 0.115.12. This means **downgrading** fastapi. However, the `requirements.txt` floor pin is `>=0.109.0`, and the installed version may vary by environment.

**Resolution:** Before pinning to 0.115.12, check what version is actually installed in the production (Railway) environment. If Railway has 0.129.0, we should instead pin to a version that:
1. Is >= 0.129.0 (to avoid downgrade)
2. Pulls in a fixed starlette

Run this check:
```bash
./venv/bin/pip install fastapi==0.135.3 --dry-run
```

If fastapi 0.135.3 (latest) pulls in starlette >= 0.45.3, use:
```
fastapi==0.135.3
```

**Decision tree:**
- If `fastapi==0.135.3` installs starlette >= 0.45.3 -> use `fastapi==0.135.3`
- If `fastapi==0.135.3` installs starlette < 0.45.3 -> use `fastapi==0.135.3` AND add `starlette>=0.45.3` explicitly
- If `fastapi==0.135.3` fails to install (dependency conflicts) -> try `fastapi==0.129.0` and add `starlette>=0.45.3` explicitly

**CVE verification after update:**
- CVE-2026-33010 (CORS misconfiguration): Run `./venv/bin/pip show starlette` and confirm version >= 0.45.3. Resolved.

#### 1c. Update `uvicorn` (paired with fastapi)

**Line 35 -- current:**

**Change to:**
```

```

**Why:** Pin to exact version for reproducibility. Use 0.34.0 (latest stable) since we are already updating the ASGI stack. Uvicorn 0.34.0 is compatible with both starlette 0.x and 1.x.

**Breaking change risk:** LOW. Uvicorn maintains backward compatibility. The project uses standard uvicorn invocation (`uvicorn src.api.app:app`).

#### 1d. Update `requests` (Finding #7 -- MEDIUM)

**Line 13 -- current:**
```
httpx==0.28.1
```

Note: `requests` is not currently listed in `requirements.txt` but is installed as a transitive dependency (via `google-auth`, `google-api-python-client`, etc.). The CVE applies to `requests` itself.

**Add after line 13 (after the httpx line):**
```
requests==2.33.1
```

**Why add explicitly:** The `requests` package is currently pulled in transitively and could be any version >= 2.25.0 depending on when `pip install` runs. Pinning it explicitly ensures the CVE fix is always present.

**Breaking change risk:** LOW. This is a minor version bump (2.32.5 -> 2.33.1). The `requests` library has excellent backward compatibility. The project does not call `requests.utils.extract_zipped_paths` directly, but pinning the safe version is defensive.

**CVE verification after update:**
- CVE-2026-25645 (file replacement): Fixed in 2.33.1. Version 2.33.1 >= 2.33.1. Resolved.

## Stress Testing & Edge Cases

### Edge Cases

1. **FastAPI version conflict:** If `fastapi==0.135.3` conflicts with other packages (e.g., `httpx` version constraints), follow the decision tree in Step 1b. The fallback is to keep the current fastapi version and explicitly pin `starlette>=0.45.3`.

2. **Starlette major version mismatch:** FastAPI 0.115.x-0.135.x depends on starlette 0.x. Do NOT manually install starlette 1.x -- it will break FastAPI. Always let FastAPI determine the starlette version.

3. **npm peer dependency warnings:** The Next.js update may produce peer dependency warnings for `eslint-config-next` or `@types/react`. These are non-blocking warnings, not errors. Ensure the build still succeeds.

4. **Package-lock.json merge conflicts:** If other PRs also modify `landing/package-lock.json`, there will be merge conflicts. Resolve by re-running `npm install` on the merged branch -- never manually edit `package-lock.json`.

5. **Railway deployment:** After merging this PR, Railway will auto-deploy. The `requirements.txt` changes will trigger a fresh `pip install` on Railway. Verify the deployment succeeds by checking Railway logs and running:
   ```bash
   storyline-cli check-health
   ```

### Performance Considerations

None. These are security patches, not feature changes. No performance impact expected.

## What NOT To Do

1. **Do NOT update all 35 outdated Python packages at once.** This phase addresses only the 4 packages with known CVEs. Updating everything in a single PR makes it impossible to isolate regressions. Non-security updates belong in Phase 05 (dependency hygiene).

2. **Do NOT use `>=` pins for the updated packages.** Pin to exact versions with `==`. The research identified inconsistent pinning as a security risk. This phase begins the migration to exact pinning for security-critical packages.

3. **Do NOT run `npm audit fix --force` blindly.** The `--force` flag can make major version jumps that break the build. First try `npm audit fix` (without `--force`). Only use `--force` after verifying the forced changes do not break the build, and only for specific packages.

4. **Do NOT manually install starlette 1.x.** FastAPI controls its starlette dependency. Installing starlette 1.x alongside FastAPI 0.x will cause runtime crashes. Always let FastAPI's dependency resolver pick the starlette version.

5. **Do NOT skip the `pip-audit` verification step.** The whole point of this phase is to eliminate known CVEs. Without `pip-audit`, you cannot confirm the CVEs are actually resolved.

6. **Do NOT edit `landing/package-lock.json` by hand.** Always regenerate it via `npm install`. Manual edits to lock files cause subtle, hard-to-debug dependency resolution issues.

7. **Do NOT merge without running the full test suite.** Even "trivial" version bumps can break things -- a changed default, a deprecated API removal, or a transitive dependency conflict. The test suite is the safety net.

8. **Do NOT update `drizzle-orm` in this phase.** The research mentions a drizzle-orm SQL injection (GHSA-gpj5-g38j-94v9), but the task scope explicitly covers only the 5 findings listed. Drizzle-orm is handled separately if scoped in another phase.
